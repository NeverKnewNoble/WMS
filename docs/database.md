# Warehouse MS — PostgreSQL Database Design

This document is the source of truth for the relational model that backs the
Warehouse MS portal. Every page in `src/app/portal/**` and the auth flows in
`src/app/auth/**` should map cleanly onto the tables, views, and queries
described here.

- **Engine:** PostgreSQL 15+
- **Naming:** `snake_case` for tables, columns, constraints, and indexes
- **IDs:** `BIGINT` surrogate keys generated with `GENERATED BY DEFAULT AS IDENTITY`
- **Timestamps:** every table carries `created_at` / `updated_at` (`TIMESTAMPTZ`, default `now()`)
- **Soft delete:** mutable business entities use `deleted_at TIMESTAMPTZ`; hard deletes are reserved for join tables
- **Money:** `NUMERIC(14,2)` (Ghana cedi by default)
- **Quantities:** `NUMERIC(14,3)` so we can store `0.5` bags or `0.250` litres without rounding
- **Email:** `CITEXT` for case-insensitive uniqueness

## 1. ER overview

```
                     |    users     |
                     +--------------+
                       ^         ^
                       |         |
                       |         |
+-----------+      +---+---+   +-+----------------+      +-----------+
|  roles    |<-----+ users |   | user_sessions    |      |  regions  |
+-----------+      +---+---+   +------------------+      +-----+-----+
                       |                                       |
                       |                                       |
                       v                                       v
                 +-----+--------+                        +-----+-----+
                 | departments  |                        | projects  |
                 +--------------+                        +-----+-----+
                                                              |
                                                              v
                  +---------------+        +-----------+   +--+--------------+
                  |  categories   |        | suppliers |   | stock_movements |
                  +-------+-------+        +-----+-----+   +--+--------------+
                          |                      |            |
                          v                      v            v
                       +--+----+              +--+---+     +--+----------------+
                       | items |--+         (movement |    | stock_movement_   |
                       +-------+  |          counter- |    |   items           |
                                  |          parties) |    +-------------------+
                                  v                                  |
                              +---+---+                              |
                              | units |<-----------------------------+
                              +-------+
```

Two parallel “movement” flows run through the same tables: **operations**
(GRN, MRN against a `project`) and **maintenance** (against a `site` /
`technician` / `application`). Both are unified in `stock_movements` and
distinguished by `kind`.

```sql
CREATE TYPE item_status        AS ENUM ('in_stock', 'low', 'critical', 'out');

CREATE EXTENSION IF NOT EXISTS pgcrypto; -- for gen_random_uuid()
```

`item_status` is **derived** in a view (see §9) — never stored. We keep an
explicit `out` value because UI also shows a zero-stock pill (the
`StatusTone` enum in `primitives.tsx`).

## 3. Reference / lookup tables

These exist so the UI's free-text dropdowns become real foreign keys.

```sql
CREATE TABLE roles (
  id           BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  code         TEXT NOT NULL UNIQUE,            -- 'warehouse_manager', 'storekeeper', ...
  label        TEXT NOT NULL,                   -- 'Warehouse manager'

  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()

CREATE TABLE departments (
  id           BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,

  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
```

## 6. Items (inventory registry)

Backs `src/app/portal/inventory/**`. The `current_stock` column is a
**denormalized cache** maintained by the trigger in §8; the source of truth is
`stock_movement_items`.

```sql
CREATE TABLE items (
  id                 BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  rfq                TEXT NOT NULL UNIQUE,                 -- 'RFQ-001'
  name               TEXT NOT NULL,
  category_id        BIGINT NOT NULL REFERENCES categories(id),
  unit_id            BIGINT NOT NULL REFERENCES units(id),
  default_supplier_id BIGINT REFERENCES suppliers(id),
  manufacturer_id    BIGINT REFERENCES manufacturers(id),
  is_maintenance_part BOOLEAN NOT NULL DEFAULT FALSE,
  current_stock      NUMERIC(14,3) NOT NULL DEFAULT 0,     -- denormalized cache
  reorder_level      NUMERIC(14,3) NOT NULL DEFAULT 0,
  min_stock          NUMERIC(14,3) NOT NULL DEFAULT 0,
  max_stock          NUMERIC(14,3) NOT NULL DEFAULT 0,
  description        TEXT,
  created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at         TIMESTAMPTZ,
  CHECK (reorder_level >= min_stock),
  CHECK (max_stock     >= reorder_level)
);

CREATE INDEX idx_items_category        ON items(category_id);
CREATE INDEX idx_items_supplier        ON items(default_supplier_id);
CREATE INDEX idx_items_alive           ON items(deleted_at) WHERE deleted_at IS NULL;
CREATE INDEX idx_items_name_trgm       ON items USING gin (name gin_trgm_ops);
CREATE INDEX idx_items_low_stock       ON items(current_stock)
  WHERE deleted_at IS NULL AND current_stock <= reorder_level;

CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

## 7. Stock movements (unified ledger)

Backs the **Stock in**, **Stock out**, and **Maintenance** pages plus the
**Stock movement history** report. A single header table keeps the ledger
sortable in one query; `kind` distinguishes operations vs. maintenance and
controls which columns are required.

```sql
CREATE TABLE stock_movements (
  id                       BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  ref_no                   TEXT NOT NULL UNIQUE,           -- 'GRN-26-001', 'MRN-2026-001', 'MAINT-IN-001'
  direction                movement_direction NOT NULL,
  kind                     movement_kind      NOT NULL,
  movement_date            DATE NOT NULL,
  rfq                      TEXT,                           -- procurement reference
  notes                    TEXT,

  -- Operations stock-in
  supplier_id              BIGINT REFERENCES suppliers(id),
  received_by_user_id      BIGINT REFERENCES users(id),

  -- Operations stock-out
  project_id               BIGINT REFERENCES projects(id),
  department_id            BIGINT REFERENCES departments(id),
  activity                 TEXT,                           -- 'Foundation works'
  issued_to_user_id        BIGINT REFERENCES users(id),    -- recipient
  authorised_by_user_id    BIGINT REFERENCES users(id),

  -- Maintenance
  site_id                  BIGINT REFERENCES sites(id),
  technician_user_id       BIGINT REFERENCES users(id),
  application              TEXT,                           -- 'Generator Repair'
  manufacturer_id          BIGINT REFERENCES manufacturers(id),
  storage_location_id      BIGINT REFERENCES storage_locations(id),

  created_at               TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at               TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_by_user_id       BIGINT REFERENCES users(id),
  deleted_at               TIMESTAMPTZ,

  -- Operations stock-in must have a supplier; stock-out must have a project.
  CHECK (
    (kind = 'operations' AND direction = 'in'  AND supplier_id IS NOT NULL)
    OR (kind = 'operations' AND direction = 'out' AND project_id  IS NOT NULL)
    OR (kind = 'maintenance' AND site_id IS NOT NULL)
  )
);

CREATE INDEX idx_movements_kind_dir_date
  ON stock_movements (kind, direction, movement_date DESC)
  WHERE deleted_at IS NULL;
CREATE INDEX idx_movements_project ON stock_movements(project_id);
CREATE INDEX idx_movements_supplier ON stock_movements(supplier_id);
CREATE INDEX idx_movements_site     ON stock_movements(site_id);
CREATE INDEX idx_movements_ref_trgm ON stock_movements USING gin (ref_no gin_trgm_ops);


CREATE TABLE stock_movement_items (
  id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  movement_id     BIGINT NOT NULL REFERENCES stock_movements(id) ON DELETE CASCADE,
  item_id         BIGINT NOT NULL REFERENCES items(id),
  qty             NUMERIC(14,3) NOT NULL CHECK (qty > 0),
  unit_id         BIGINT NOT NULL REFERENCES units(id),
  unit_cost       NUMERIC(14,2),                            -- optional, for valuation
  condition       movement_condition NOT NULL DEFAULT 'good',
  note            TEXT
);

CREATE INDEX idx_movement_items_movement ON stock_movement_items(movement_id);
CREATE INDEX idx_movement_items_item     ON stock_movement_items(item_id);
```

> **Why the `CHECK` is wide rather than per-column NOT NULL:** the same table
> serves three flows; the constraint encodes which columns each flow requires
> without forcing every column to be NOT NULL.

## 8. Triggers — keep `items.current_stock` honest

```sql
CREATE OR REPLACE FUNCTION apply_movement_to_stock()
RETURNS TRIGGER AS $$
DECLARE
  v_direction movement_direction;
  v_alive     BOOLEAN;
BEGIN
  -- Only mutate stock for non-deleted movements.
  IF TG_OP = 'INSERT' THEN
    SELECT direction, deleted_at IS NULL
      INTO v_direction, v_alive
      FROM stock_movements WHERE id = NEW.movement_id;

    IF v_alive THEN
      UPDATE items SET
        current_stock = current_stock
          + CASE WHEN v_direction = 'in' THEN NEW.qty ELSE -NEW.qty END,
        updated_at = now()
      WHERE id = NEW.item_id;
    END IF;

  ELSIF TG_OP = 'DELETE' THEN
    SELECT direction, deleted_at IS NULL
      INTO v_direction, v_alive
      FROM stock_movements WHERE id = OLD.movement_id;

    IF v_alive THEN
      UPDATE items SET
        current_stock = current_stock
          - CASE WHEN v_direction = 'in' THEN OLD.qty ELSE -OLD.qty END,
        updated_at = now()
      WHERE id = OLD.item_id;
    END IF;
  END IF;

  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_movement_items_stock
AFTER INSERT OR DELETE ON stock_movement_items
FOR EACH ROW EXECUTE FUNCTION apply_movement_to_stock();

-- Reverse the cache when a movement is soft-deleted, replay it when restored.
CREATE OR REPLACE FUNCTION apply_movement_soft_delete()
RETURNS TRIGGER AS $$
DECLARE
  sign INT;
BEGIN
  IF (OLD.deleted_at IS NULL) AND (NEW.deleted_at IS NOT NULL) THEN
    sign := -1;
  ELSIF (OLD.deleted_at IS NOT NULL) AND (NEW.deleted_at IS NULL) THEN
    sign := 1;
  ELSE
    RETURN NEW;
  END IF;

  UPDATE items i
  SET current_stock = i.current_stock
        + sign * CASE WHEN NEW.direction = 'in' THEN smi.qty ELSE -smi.qty END,
      updated_at = now()
  FROM stock_movement_items smi
  WHERE smi.movement_id = NEW.id AND smi.item_id = i.id;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_movement_soft_delete
AFTER UPDATE OF deleted_at ON stock_movements
FOR EACH ROW EXECUTE FUNCTION apply_movement_soft_delete();
```

The two triggers together give safe behavior for the **delete confirmation
modals** in the UI: a soft-deleted GRN/MRN reverses its effect on stock, and
restoring it puts the effect back.

## 9. Views — what the UI actually reads

### 9.1 `v_items_with_status`

Drives the **Inventory** table and the status pills.

```sql
CREATE VIEW v_items_with_status AS
SELECT
  i.id,
  i.rfq,
  i.name,
  c.label AS category,
  u.label AS unit,
  i.current_stock,
  i.reorder_level,
  i.min_stock,
  i.max_stock,
  CASE
    WHEN i.current_stock <= 0                  THEN 'out'
    WHEN i.current_stock <= i.min_stock        THEN 'critical'
    WHEN i.current_stock <= i.reorder_level    THEN 'low'
    ELSE 'in_stock'
  END::item_status AS status,
  i.is_maintenance_part,
  i.deleted_at IS NULL AS is_active
FROM items i
JOIN categories c ON c.id = i.category_id
JOIN units      u ON u.id = i.unit_id;
```

### 9.2 `v_reorder_alerts`

Drives `src/app/portal/reorder-alerts`.

```sql
CREATE VIEW v_reorder_alerts AS
SELECT
  i.id                AS item_id,
  i.name,
  c.label             AS category,
  i.current_stock     AS current,
  i.reorder_level     AS reorder,
  GREATEST(0, i.reorder_level - i.current_stock) AS shortfall,
  GREATEST(i.reorder_level * 2, i.max_stock)     AS suggested,
  s.name              AS supplier,
  CASE
    WHEN i.current_stock <= 0                  THEN 'critical'
    WHEN i.current_stock <= i.min_stock        THEN 'critical'
    WHEN i.current_stock <= i.reorder_level    THEN 'low'
    WHEN i.current_stock <= i.reorder_level * 1.15 THEN 'watch'
    ELSE NULL
  END::alert_severity AS severity
FROM items i
JOIN categories c        ON c.id = i.category_id
LEFT JOIN suppliers s    ON s.id = i.default_supplier_id
WHERE i.deleted_at IS NULL
  AND i.current_stock <= i.reorder_level * 1.15;
```

### 9.3 `v_project_consumption`

Drives the **Projects** table and **Material consumption per project** report.

```sql
CREATE VIEW v_project_consumption AS
SELECT
  p.id              AS project_id,
  p.wbs,
  p.name,
  p.location,
  COUNT(DISTINCT smi.item_id) FILTER (WHERE m.direction = 'out') AS items_issued,
  COALESCE(SUM(smi.qty)        FILTER (WHERE m.direction = 'out'), 0) AS qty_consumed,
  MAX(m.movement_date)         FILTER (WHERE m.direction = 'out') AS last_activity
FROM projects p
LEFT JOIN stock_movements      m   ON m.project_id = p.id AND m.deleted_at IS NULL
LEFT JOIN stock_movement_items smi ON smi.movement_id = m.id
WHERE p.deleted_at IS NULL
GROUP BY p.id;
```

### 9.4 `v_dashboard_kpis`

Drives the dashboard at `src/app/portal/page.tsx`.

```sql
CREATE VIEW v_dashboard_kpis AS
SELECT
  (SELECT SUM(current_stock) FROM items WHERE deleted_at IS NULL)
    AS total_in_stock,

  (SELECT COUNT(*) FROM items
    WHERE deleted_at IS NULL AND current_stock <= reorder_level)
    AS items_below_reorder,

  COALESCE((
    SELECT SUM(smi.qty)
    FROM stock_movements m
    JOIN stock_movement_items smi ON smi.movement_id = m.id
    WHERE m.deleted_at IS NULL
      AND m.direction = 'in'
      AND m.movement_date = CURRENT_DATE
  ), 0) AS today_stock_in,

  COALESCE((
    SELECT SUM(smi.qty)
    FROM stock_movements m
    JOIN stock_movement_items smi ON smi.movement_id = m.id
    WHERE m.deleted_at IS NULL
      AND m.direction = 'out'
      AND m.movement_date = CURRENT_DATE
  ), 0) AS today_stock_out;
```

### 9.5 `v_stock_movement_history`

Backs the **Stock movement history** report and the maintenance unified ledger.

```sql
CREATE VIEW v_stock_movement_history AS
SELECT
  m.id              AS movement_id,
  m.ref_no,
  m.direction,
  m.kind,
  m.movement_date,
  smi.qty,
  u.label           AS unit,
  it.name           AS item,
  c.label           AS category,
  s.name            AS supplier,
  p.name            AS project,
  p.wbs             AS project_wbs,
  d.label           AS department,
  st.label          AS site,
  tech.full_name    AS technician,
  m.application,
  m.activity,
  m.rfq
FROM stock_movements m
JOIN stock_movement_items smi ON smi.movement_id = m.id
JOIN items it      ON it.id = smi.item_id
JOIN categories c  ON c.id = it.category_id
JOIN units u       ON u.id = smi.unit_id
LEFT JOIN suppliers s   ON s.id = m.supplier_id
LEFT JOIN projects p    ON p.id = m.project_id
LEFT JOIN departments d ON d.id = m.department_id
LEFT JOIN sites st      ON st.id = m.site_id
LEFT JOIN users tech    ON tech.id = m.technician_user_id
WHERE m.deleted_at IS NULL;
```

### 9.6 `v_slow_moving_items`

Backs the **Slow-moving stock** report.

```sql
CREATE VIEW v_slow_moving_items AS
SELECT
  i.id,
  i.rfq,
  i.name,
  c.label AS category,
  i.current_stock,
  MAX(m.movement_date) FILTER (WHERE m.direction = 'out') AS last_issued,
  CURRENT_DATE
    - COALESCE(MAX(m.movement_date) FILTER (WHERE m.direction = 'out'), i.created_at::date)
    AS days_idle
FROM items i
JOIN categories c                  ON c.id = i.category_id
LEFT JOIN stock_movement_items smi ON smi.item_id = i.id
LEFT JOIN stock_movements m        ON m.id = smi.movement_id AND m.deleted_at IS NULL
WHERE i.deleted_at IS NULL
GROUP BY i.id, c.label
HAVING (CURRENT_DATE
        - COALESCE(MAX(m.movement_date) FILTER (WHERE m.direction = 'out'),
                   i.created_at::date)) >= 30;
```

## 10. Notifications (Settings → Notification preferences)

```sql
CREATE TABLE notification_preferences (
  user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  key         TEXT   NOT NULL,                              -- 'low_stock_alert', 'stock_in_received', ...
  enabled     BOOLEAN NOT NULL DEFAULT TRUE,

CREATE TABLE notifications (
  id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  type        TEXT   NOT NULL,                              -- matches preferences.key
  title       TEXT   NOT NULL,
  body        TEXT,
  payload     JSONB  NOT NULL DEFAULT '{}'::jsonb,          -- e.g. { "item_id": 12 }
  read_at     TIMESTAMPTZ,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_notifications_user_unread
  ON notifications(user_id, created_at DESC)
  WHERE read_at IS NULL;
```

```sql
  :movement_id,
  (SELECT id FROM items WHERE rfq = $10),
```

### 13.6 Record a stock-out / MRN (modal: `add-stock-out.tsx`)

```sql
  it.name AS material,

  SUM(smi.qty) AS qty
FROM stock_movements      m

JOIN units un ON un.id = smi.unit_id

  AND m.deleted_at IS NULL

  m.ref_no AS mrn, m.movement_date AS date,
```

```sql

SELECT
  to_char(d::date, 'Dy') AS day,
  COALESCE(SUM(smi.qty) FILTER (WHERE m.direction = 'in'),  0) AS in_qty,
  COALESCE(SUM(smi.qty) FILTER (WHERE m.direction = 'out'), 0) AS out_qty
```

## 14. Search performance

Most table-views in the UI ship with a search box. Two indexes cover the

```sql
-- already declared above
CREATE INDEX idx_items_name_trgm     ON items             USING gin (name   gin_trgm_ops);
CREATE INDEX idx_movements_ref_trgm  ON stock_movements   USING gin (ref_no gin_trgm_ops);

-- multicolumn for the global search bar in the dashboard
CREATE INDEX idx_projects_search
  ON projects USING gin (
    (coalesce(wbs,'') || ' ' || coalesce(name,'') || ' ' || coalesce(location,'')) gin_trgm_ops
  );
```

## 16. Open questions / future work

- **RFQ lifecycle:** the UI treats RFQ numbers as freeform text. If
  procurement workflow is added, promote `rfq` to its own table with a
  status enum (`draft`, `sent`, `received`, `closed`).
- **Multi-tenant:** if more than one company runs on this database, add a
  `tenant_id` to every business table and a row-level security policy.
- **Inventory valuation:** `unit_cost` is captured per movement, so FIFO
  and weighted-average can be computed later without backfill.
- **Real-time:** stream changes from `stock_movements` and `notifications`
  via `LISTEN/NOTIFY` or logical replication if the portal grows real-time
  features.
