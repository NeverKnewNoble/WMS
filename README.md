# Warehouse MS

A warehouse management portal for tracking inventory, stock movements,
projects, maintenance parts, and reorder alerts. Built with Next.js 16, React
19, Tailwind 4, Prisma 7, Auth.js v5, and a thin typed service layer.

---

## Stack

| Layer       | Choice                                                                  |
| ----------- | ----------------------------------------------------------------------- |
| Framework   | Next.js 16 (App Router, Turbopack, React Compiler)                       |
| Language    | TypeScript                                                              |
| Styling     | Tailwind CSS v4, lucide-react                                            |
| Database    | PostgreSQL 15+                                                          |
| ORM         | Prisma 7 (`prisma-client` generator → `src/generated/prisma`)            |
| Driver      | `@prisma/adapter-pg`                                                    |
| Auth        | Auth.js v5 (Credentials provider, JWT sessions, argon2id hashing)        |
| Validation  | Zod                                                                     |
| API client  | Typed service layer in `src/services/**` over a small `fetch` wrapper    |
| Toasts      | sonner (mounted globally; surfaces every service error automatically)    |

> **Heads up:** Next 16 renamed Middleware → **Proxy**. The auth gate lives in
> `src/proxy.ts`, not `middleware.ts`. Prisma 7 also moved the connection URL
> out of `schema.prisma` into `prisma.config.ts`.

## Getting started

### 1. Prerequisites

- Node.js 20+
- Yarn 1.22 (the project's `packageManager`)
- A reachable PostgreSQL 15+ instance

### 2. Install

```bash
yarn
```

The `postinstall` hook runs `prisma generate`, which writes the Prisma client
into `src/generated/prisma`.

### 3. Configure environment

Copy the template and fill in real values:

```bash
cp .env.example .env
```

`.env` keys:

| Key            | Notes                                                                   |
| -------------- | ----------------------------------------------------------------------- |
| `DATABASE_URL` | e.g. `postgresql://<user>@localhost:5432/warehouse_ms?schema=public`. On Homebrew Postgres the user is usually your macOS username with no password. |
| `AUTH_SECRET`  | `openssl rand -base64 32`                                               |
| `AUTH_URL`     | `http://localhost:3000` in dev                                          |

> If `yarn db:migrate` returns `P1010: User was denied access`, your
> `DATABASE_URL` credentials don't match the actual Postgres install — run
> `psql -l` to see what user/database you have, then adjust the URL.

### 4. Create + migrate the database

```bash
createdb warehouse_ms        # if it doesn't exist
yarn db:migrate              # applies prisma/migrations/* (Prisma will name the first run)
yarn db:seed                 # populates roles, departments, units, etc.
```

The migrations layer in two parts:

1. **`20260430000001_init`** — generated by `prisma migrate diff`; creates
   every table, enum, and FK.
2. **`20260430000002_triggers_views_constraints`** — hand-written; adds the
   bits Prisma can't express: partial / GIN indexes, the `stock_movements`
   `kind`/`direction` CHECK constraint, the two stock-cache triggers, and the
   six read-side views (`v_items_with_status`, `v_reorder_alerts`,
   `v_project_consumption`, `v_dashboard_kpis`, `v_stock_movement_history`,
   `v_slow_moving_items`).

### 5. Run the app

```bash
yarn dev
```

Open [http://localhost:3000](http://localhost:3000). Sign up at
`/auth/signup`; the proxy will then admit you to `/portal`.

## Architecture

### Database

Read **[`docs/database.md`](docs/database.md)** first. It is the canonical
description of the schema and includes:

- ER overview
- Every table grouped by concern (lookups, identity, projects, items,
  movements, notifications, audit)
- Enums + Postgres extensions
- Triggers that keep `items.current_stock` in sync with `stock_movement_items`
  (insert / delete / soft-delete)
- Views the UI reads
- Seed data
- ~20 named queries mapped to specific UI features
- Migration strategy

The Prisma schema in `prisma/schema.prisma` mirrors §3-§7 and §10-§11 of that
document. §8 (triggers) and §9 (views) live in
`prisma/migrations/20260430000002_triggers_views_constraints/migration.sql`.

### API

Every page reads through a Route Handler in `src/app/api/**`. The route
files are thin wrappers over Prisma, with `withApi(...)` turning thrown
`HttpError`s into JSON. Read [`docs/api.md`](docs/api.md) for the full
endpoint list and request/response shapes.

### Service layer

Pages don't call `fetch`. They call typed functions like `listItems()` and
`createMovement(...)` from `src/services/**`. The service layer:

- Wraps every endpoint in a 5-line `try / catch` block.
- Pops a sonner toast (title = friendly fallback, description = server
  error) on failure.
- Throws an `ApiError` so callers can branch on `err.status` for control
  flow (e.g. redirect on 401).
- Exposes a `useService(loader, deps)` hook for read pages, with race-safe
  refetch.

Read [`docs/services.md`](docs/services.md) for the recipes and a
function-by-function reference.

### Stock movements (the unified ledger)

Operations stock-in (GRN), operations stock-out (MRN), and maintenance
movements all live in **one table** — `stock_movements` — with a `kind` enum
(`operations` / `maintenance`) and a `direction` enum (`in` / `out`). A
multi-column `CHECK` ensures each kind/direction has the columns it needs
(supplier for ops in, project for ops out, site for maintenance). Line items
go in `stock_movement_items`.

`items.current_stock` is a denormalized cache. Two triggers keep it honest:

- `trg_movement_items_stock` adjusts stock on `INSERT` / `DELETE` of a line
  item.
- `trg_movement_soft_delete` reverses the effect when a movement's
  `deleted_at` is set, and replays it if it's cleared.

This means the **delete confirmation modals** in the UI Just Work —
`deleteMovement(id)` from the service soft-deletes the row and stock
corrects itself.

### Authentication

`src/lib/auth.ts` wires Auth.js v5:

- **Credentials provider** verifies argon2id hashes from `users.password_hash`.
- **JWT sessions** (no DB session table required).
- The `jwt` / `session` callbacks propagate `userId`, `role`, and
  `departmentId` into `session.user` so server components and route handlers
  can authorize without an extra query.
- The `authorized` callback redirects unauthenticated users to `/auth/login`
  and bounces logged-in users out of `/auth/**`.

`src/proxy.ts` (Next 16's renamed Middleware) re-exports the `auth` helper and
applies a `matcher` that excludes `/api/auth/**` and static assets.

`src/app/api/auth/register/route.ts` is the signup endpoint: zod-validates the
payload, argon2-hashes the password, defaults the new user to the
`storekeeper` role, and returns the created user. The signup page then calls
`signIn("credentials", …)` to log them in immediately.

### UI patterns

- **Modals** live in `src/components/modals` and use `createPortal`. Two
  shapes: self-contained (own button + state, e.g. `add-item.tsx`) and
  controlled (parent owns the open state, e.g. `edit-item.tsx`,
  `project-details.tsx`, `report-viewer.tsx`, `confirm-delete.tsx`).
- **`confirm-delete.tsx`** is generic: pass a `title`, `message`, `details`
  array, and `onConfirm`. Reused by projects, stock-in, stock-out, and
  maintenance.
- **Tables** use `w-full min-w-max` inside an `overflow-x-auto` wrapper so
  columns render at their natural width and the wrapper scrolls horizontally
  rather than squeezing.
- **Primitives** (`PageHeader`, `Surface`, `StatusPill`, `MonoCell`,
  `FieldLabel`, `fieldClass`) live in
  `src/components/ui_components/portal/primitives.tsx`. Use them rather than
  re-rolling card / input styles per page.
- **Toasts** appear bottom-right with a dark theme matching the portal.
  Every service catch already triggers them via `failWithToast(err, …)` —
  pages don't need to call `toast.error` themselves. Use `showSuccessToast`
  after successful mutations.

## Development notes

### Prisma 7 specifics

- The connection URL is **not** in `schema.prisma`. It lives in
  `prisma.config.ts` (`datasource.url = env("DATABASE_URL")`).
- The runtime client requires either an `adapter` or `accelerateUrl`. We use
  `@prisma/adapter-pg`. See `src/lib/prisma.ts`.
- We use the new `prisma-client` generator (not the legacy `prisma-client-js`)
  with `output = "../src/generated/prisma"`. This avoids the
  `.prisma/client/default` indirection that Turbopack can't bundle.
- `src/generated/` is gitignored; it's regenerated by `prisma generate`.

### Next 16 specifics

- `proxy.ts` replaces `middleware.ts`. The functionality is identical.
- `cookies()` is async — `await cookies()` before reading.
- The Prisma client and `@node-rs/argon2` are already on Next's automatic
  `serverExternalPackages` list, so no manual config needed.
- `tsconfig.target` is **ES2020** (bumped from ES2017 so `BigInt` literals
  like `0n` compile).

### Adding a new entity

The same flow appears at the bottom of `docs/services.md`. Briefly:

1. Add the model to `prisma/schema.prisma`.
2. `yarn db:migrate` (Prisma will prompt for a name and generate the SQL).
3. If the entity needs triggers / views / partial indexes, add a hand-written
   `prisma/migrations/<timestamp>_<name>/migration.sql` and update
   `docs/database.md` §8 / §9 / §14.
4. Add a route handler under `src/app/api/<thing>/route.ts` (and a
   section in `docs/api.md`).
5. Declare types in `src/types/<thing>.ts`.
6. Add a service in `src/services/<thing>.ts` — one async function per
   route, each wrapping `http.*` in a `try / failWithToast` pair. Update
   the endpoint table in `docs/services.md`.
7. Re-run `yarn db:migrate` to apply the supplementary migration.

### Adding a new modal

1. Create `src/components/modals/<name>.tsx`.
2. If parent-controlled, accept `<thing>: T | null` and `onClose: () => void`.
3. Use `createPortal(...)` and add an Escape-key listener + `body` scroll lock
   inside `useEffect`. Mirror the existing modals (`edit-item.tsx`,
   `project-details.tsx`).
